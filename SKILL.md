# Study Slicer · 学习切片器

将学习笔记 / 教程文件切片整理，生成可跨设备阅读的单页 HTML 知识手册。
支持**增量迭代**：持续向同一个 HTML 文件追加/完善知识内容。

## 触发条件

### 模式 A：首次生成

用户上传学习文件并要求生成知识手册，且目标目录下无已有手册：
- 切片、整理、提炼重点
- 生成知识手册 / 学习卡片
- 做成网页方便跨设备学习

### 模式 B：增量迭代（追加内容）

用户上传新文件/内容，要求更新已有手册：
- "把这些内容加到 XX 手册里"
- "继续完善这个手册"
- "把新笔记合并进去"
- "更新一下这个网页"
- 目标目录下已存在 `-handbook.html` 文件

**关键原则：只有一个 HTML 文件，不断迭代完善。**

## 模板变量

生成 HTML 时，用以下变量替换 `templates/handbook.html` 中的 `{{占位符}}`：

| 变量 | 说明 | 示例 |
|------|------|------|
| `{{TITLE}}` | 页面 `<title>` | `Java 并发编程手册` |
| `{{NAV_BRAND}}` | 导航栏左侧品牌文字 | `📚 Study Slicer` |
| `{{NAV_LINKS}}` | 导航链接，`<li><a>` 列表 | `<li><a href="#s1">线程基础</a></li>...` |
| `{{TOC_ITEMS}}` | 侧边目录条目 | 见下方格式 |
| `{{HERO_EMOJI}}` | Hero 区大 emoji | `⚡` |
| `{{HERO_TITLE}}` | Hero 大标题 | `Java 并发编程` |
| `{{HERO_DESC}}` | 一句话描述 | `从线程基础到并发框架全解析` |
| `{{HERO_STATS}}` | 统计卡片 HTML | 见下方格式 |
| `{{ROADMAP}}` | 学习路线图 | 见下方格式 |
| `{{SECTIONS}}` | 全部内容 section 的 HTML | 核心内容区 |
| `{{CHANGELOG}}` | 变更日志条目 | 见下方格式 |
| `{{FOOTER}}` | 页脚文字 | `Generated by Study Slicer · 2026-04-27` |

### HERO_STATS 格式

```html
<div class="hero-stat"><div class="num">6</div><div class="label">模块</div></div>
<div class="hero-stat"><div class="num">24</div><div class="label">知识点</div></div>
<div class="hero-stat"><div class="num">12</div><div class="label">代码示例</div></div>
<div class="hero-stat"><div class="num">8</div><div class="label">对比表格</div></div>
```

根据实际内容统计数量，生成对应 HTML。标签可灵活调整。

### TOC_ITEMS 格式（侧边目录）

为每个 section 生成一条目，格式如下：

```html
<a href="#s1" class="toc-item"><span class="toc-num">01</span>线程基础</a>
<a href="#s2" class="toc-item"><span class="toc-num">02</span>线程同步</a>
<a href="#s3" class="toc-item"><span class="toc-num">03</span>线程池</a>
```

编号与 section 的 `id` 和 `.section-num` 对应。

### ROADMAP 格式（学习路线图）

**⚡ 动态演进：每次迭代时必须基于全部已有内容重新生成路线图，而非简单追加。**

根据当前手册的**所有模块**重新规划推荐学习顺序，用流程图形式展示：

```html
<div class="roadmap">
  <div class="rm-step active"><div class="rm-dot">1</div><div class="rm-label">基础语法</div></div>
  <div class="rm-arrow">→</div>
  <div class="rm-step"><div class="rm-dot">2</div><div class="rm-label">面向对象</div></div>
  <div class="rm-arrow">→</div>
  <div class="rm-step"><div class="rm-dot">3</div><div class="rm-label">集合框架</div></div>
  <div class="rm-arrow">→</div>
  <div class="rm-step"><div class="rm-dot">4</div><div class="rm-label">多线程</div></div>
</div>
```

**路线图生成规则（每次迭代都要执行）：**
- **丢弃旧路线图，基于全部模块重新生成**
- 按知识依赖关系排列顺序（前置在前，进阶在后）
- 新内容改变了模块间依赖 → 调整顺序
- 新内容让某模块不再核心 → 降级或合并
- 新内容覆盖了某模块的前置 → 插入新步骤
- 新模块加入 → 找到它在依赖链中的正确位置插入
- `rm-step` 控制在 4~8 个（太多就合并相关模块）
- `rm-arrow` 用 `→` 连接（移动端自动换行）

### CHANGELOG 格式（变更日志）

每次更新手册时，在 changelog 区域**顶部插入**一条新记录（最新的在最上面）：

```html
<div class="cl-entry">
  <div class="cl-date">2026-04-27</div>
  <div class="cl-desc">初始版本：6 个模块，24 个知识点，12 个代码示例</div>
</div>
<div class="cl-entry">
  <div class="cl-date">2026-04-28</div>
  <div class="cl-desc">新增「线程池」模块 3 张卡片、「并发工具」模块 2 张卡片；补充 CompletableFuture 代码示例</div>
</div>
```

首次生成时只有一条。迭代时在顶部插入新条目，保留所有历史记录。

## 工作流程

### Step 1：获取内容

```
用户输入来源：
├── 本地文件路径 → read 工具读取
├── URL 链接 → web_fetch 抓取
├── 附件上传 → 读取 mimo-files 中的 URL
├── 图片截图 → 调用 mimo-omni skill 识别文字（OCR）
└── 直接粘贴文本 → 直接使用
```

- 如果是 URL，用 `web_fetch` 下载内容
- 如果是本地文件，用 `read` 读取
- 如果是图片（.png/.jpg），用 `mimo-omni` 进行 OCR 识别
- 如果是 PDF，优先提取文本；若为扫描件则用 OCR
- 如果内容超过 50KB，分段读取后拼接
- 如果是 YouTube 链接，先用 `summarize` skill 提取字幕/摘要

### Step 2：检测已有手册

**在工作目录中搜索 `-handbook.html` 文件：**

```bash
find /root/.openclaw/workspace -name "*-handbook.html" -type f 2>/dev/null
```

- **找到** → 进入增量迭代模式（Step 2B）
- **未找到** → 进入首次生成模式（Step 2A）

---

### Step 2A：首次生成模式

对读取到的内容进行结构化分析：

**识别维度：**
1. **主题模块** — 按大章节 / 知识域划分（如"线程创建"、"线程状态"）
2. **知识点** — 每个模块下的核心概念、原理、用法
3. **代码示例** — 提取所有代码块，保留语言标注
4. **对比关系** — 识别表格、对比、区别类内容
5. **流程/生命周期** — 识别步骤、状态流转等
6. **易错点/面试题** — 识别"注意"、"陷阱"、"常见错误"等
7. **难度分级** — 每张卡片标注难度等级
8. **前置依赖** — 标注知识点间的前后依赖关系

**切片原则：**
- 每个模块自成体系，可独立阅读
- 模块间有逻辑递进关系
- 代码和解释配对，不分离
- 对比内容用表格呈现
- 流程内容用步骤图呈现
- 提取类比/比喻帮助理解

然后进入 Step 3 生成全新 HTML。

---

### Step 2B：增量迭代模式

**核心：读取已有 HTML → 解析结构 → 合并新内容 → 写回同一文件。**

#### 2B.1 读取并解析已有手册

用 `read` 工具读取已有的 `-handbook.html` 文件，提取以下信息：

```
解析目标：
├── 当前标题（<title> 和 Hero 区）
├── 当前导航链接列表（所有 <li><a href="#sN">）
├── 当前侧边目录（所有 toc-item）
├── 当前学习路线图（rm-step）
├── 当前所有 section 的 id 和标题
├── 每个 section 内的卡片（.card）标题和内容摘要
├── 每张卡片的难度标签（.diff）和来源标注（.source）
├── 当前统计数字（hero-stats）
├── 当前变更日志（cl-entry 条目）
└── 文件路径（用于最终写回）
```

#### 2B.2 分析新内容

对新输入的内容执行同样的结构化分析（同 Step 2A 的识别维度）。

#### 2B.3 智能合并策略

将新内容与已有结构进行匹配，决定合并方式：

```
合并决策树：
├── 新内容属于已有模块 → 追加卡片到对应 section
│   ├── 有重复知识点 → 更新/补充已有卡片（不删除原有内容）
│   └── 无重复 → 直接追加新卡片
├── 新内容是全新主题 → 创建新 section
│   ├── 插入到逻辑合理的位置（不是简单追加到末尾）
│   └── 更新导航链接、侧边目录、学习路线图
├── 新内容补充了对比关系 → 在已有卡片中添加表格
├── 新内容是易错点 → 追加到对应模块的踩坑区
└── 新内容是练习题 → 追加到对应模块的练习区
```

#### 2B.4 全量重算（路线图 + 前置知识）

**每次迭代的最后必须执行此步骤，确保路线图和前置知识随内容演进。**

**路线图重算：**
1. 收集当前手册的所有 section（包括新增和已有）
2. 分析模块间的依赖关系（哪个是前置，哪个是进阶）
3. 重新排列路线图顺序
4. 丢弃旧的 `{{ROADMAP}}` 内容，写入新的路线图
5. 如果模块超过 8 个，合并相关模块为更高层级的主题

**前置知识回溯更新：**
1. 遍历所有已有卡片，检查是否需要补充前置引用
2. 新内容可能揭示隐性依赖（例：新增"泛型"→ 集合卡片需要前置引用泛型）
3. 更新已有卡片的 `.prereq` div（如果需要的话）
4. 检查是否有循环依赖，如有则重新组织模块划分

**合并原则：**
- **只增不删**：保留已有所有内容，新内容追加或补充
- **去重但不丢信息**：如果新旧内容高度重复，合并为更完整的版本
- **保持结构连贯**：新 section 编号接续已有编号
- **更新统计**：重新计算 hero-stats 中的数字
- **更新导航**：nav-links 和 toc-items 必须包含所有 section
- **动态路线图**：每次全量重算，不追加，重新规划最优顺序
- **动态前置**：新卡片必须标注前置，已有卡片回溯补充
- **更新变更日志**：在 changelog 顶部插入本次更新记录

#### 2B.5 生成合并后的 HTML

基于解析到的已有 HTML 结构，将新内容插入正确位置，生成完整的更新版 HTML。

**不要重新生成整个文件！** 而是：
1. 复制已有 HTML 的完整结构
2. 在 `<main>` 内的正确位置插入新 section 或新 card
3. 更新已有卡片的 `.prereq`（如果回溯发现了新的依赖）
4. 更新 `<ul class="nav-links">` 中的导航
5. 更新侧边目录 `toc-items`
6. **全量重算**学习路线图 `roadmap`（丢弃旧的，重新生成）
7. 更新 `hero-stats` 中的数字
8. 在 changelog 顶部插入新条目
9. 更新 `<title>` 如果主题范围扩大了

---

### Step 3：生成 / 更新 HTML 手册

使用 `templates/handbook.html` 作为模板基础（仅首次生成时读取模板，迭代时基于已有 HTML）。

**必须包含的结构：**

```
HTML 手册结构：
├── 阅读进度条（顶部细线）
├── 侧边目录（桌面端固定左侧，移动端隐藏）
├── 顶部导航栏（固定）
│   ├── 品牌标题
│   ├── 🔍 搜索框（全文搜索，Ctrl+K 聚焦）
│   ├── 🌓 暗亮切换按钮（快捷键 T）
│   ├── 模块导航链接（自动生成）
│   └── 移动端汉堡菜单
├── Hero 区域
│   ├── 标题 + 描述
│   ├── 统计数据（模块数、知识点数等）
│   └── 学习路线图（推荐学习顺序）
├── 内容区（按模块分 section，可折叠）
│   ├── section-header（折叠开关 + 编号 + 标题）
│   ├── 知识卡片 (.card)
│   │   ├── 卡片标题 + 难度标签
│   │   ├── 前置知识链接（.prereq）
│   │   ├── 来源标注 (.source)
│   │   ├── 概念说明
│   │   ├── 代码块（带语法高亮 + 复制按钮）
│   │   ├── 对比表格
│   │   ├── 流程图 / 状态图
│   │   └── 重点提示 (.callout)
│   ├── 踩坑区 (.pitfall)
│   │   └── 初学者易犯错误
│   └── 模块内练习题（可选）
├── 速查表（汇总所有方法/概念）
├── 练习区（折叠式题目）
├── 变更日志（changelog，最新在上）
└── 页脚
```

**样式规范：**
- 默认暗色主题，支持切换亮色主题（CSS 变量驱动）
- 响应式布局（手机/平板/电脑）
- 代码块带语言标签 + 一键复制按钮
- 表格带 hover 高亮
- callout 分三种：info（信息）、warn（警告）、tip（技巧）
- 练习题用 `<details>` 折叠
- 导航栏滚动时自动高亮当前章节
- 顶部阅读进度条（JS 驱动）
- 回到顶部按钮
- 打印适配（`@media print`）

**代码高亮方案：**
使用内联 CSS 类手动标记关键词（不依赖外部库）：
- `.kw` — 关键字（紫色）
- `.str` — 字符串（绿色）
- `.cmt` — 注释（灰色）
- `.type` — 类型名（黄色）
- `.num` — 数字（橙色）
- `.fn` — 方法名（蓝色）
- `.ann` — 注解（红色）

### Step 4：输出与保存

1. 将生成的 HTML 保存到工作目录
2. 文件命名：`{主题}-handbook.html`（迭代时保持同一文件名）
3. **输出变更摘要**，告知用户本次新增/更新了什么：
   ```
   📝 手册已更新：{文件路径}
   ├── 新增模块：{模块名}
   ├── 新增卡片：{数量} 个

   ├── 新增代码示例：{数量} 个
   ├── 新增踩坑点：{数量} 个
   ├── 来源标注：{数量} 条
   └── 总模块数：{N} 个
   ```
4. 如果用户需要，可以通过 canvas 展示预览

## 卡片格式

### 基础卡片（含难度标签 + 来源 + 前置知识）

```html
<div class="card">
  <h3>📌 概念标题 <span class="diff basic">🟢 基础</span></h3>
  <div class="source">📖 来源：文件名.pdf · 第3章</div>
  <div class="prereq">🔗 前置：<a href="#s1">§1 变量与数据类型</a></div>
  <!-- 卡片内容 -->
</div>
```

### 难度标签

每张卡片**必须包含难度标签**，用以下 class：

| 标签 | class | 说明 |
|------|-------|------|
| 🟢 基础 | `.diff.basic` | 初学者必学，入门门槛 |
| 🟡 进阶 | `.diff.advanced` | 需要一定基础，深入理解 |
| 🔴 面试高频 | `.diff.interview` | 面试常考，重点关注 |
| ⚪ 了解 | `.diff.know` | 知道即可，非核心 |

### 前置知识链接

**⚡ 动态演进：每次迭代时，新卡片必须标注前置，同时回溯更新已有卡片的前置关系。**

如果某张卡片依赖其他知识点，用 `.prereq` 标注并链接到对应 section：

```html
<div class="prereq">🔗 前置：<a href="#s2">§2 接口与抽象类</a>、<a href="#s3">§3 多态</a></div>
```

**前置知识更新规则（每次迭代都要执行）：**

1. **新卡片**：必须检查它依赖哪些已有知识点，标注 `.prereq`
2. **已有卡片回溯**：新内容可能揭示已有卡片的隐性前置
   - 例：新增了"泛型"模块 → 回溯检查"集合"卡片，补充 `🔗 前置：§X 泛型`
   - 例：新增了"异常处理"模块 → 回溯检查"IO"卡片，补充前置
3. **依赖链缩短**：如果新内容合并了两个概念，简化引用
   - 例：新增"面向对象总览" → 原来引用 §3 和 §4 的卡片改为引用新模块
4. **循环依赖检测**：如果 A 依赖 B、B 又依赖 A，说明分类有问题，重新组织

如果没有前置依赖，省略此 div。

### 踩坑区格式

每个 section 底部可放一个踩坑区，集中展示该模块的常见错误：

```html
<div class="pitfall">
  <div class="pitfall-title">⚠️ 踩坑区 · 初学者必看</div>
  <div class="pitfall-item">
    <div class="pitfall-wrong">❌ 错误：<code>String s = new String("abc");</code></div>
    <div class="pitfall-right">✅ 正确：<code>String s = "abc";</code></div>
    <div class="pitfall-why">原因：new 会创建额外对象，浪费内存。直接字面量赋值更高效。</div>
  </div>
  <div class="pitfall-item">
    <div class="pitfall-wrong">❌ 错误：用 == 比较字符串内容</div>
    <div class="pitfall-right">✅ 正确：用 .equals() 比较</div>
    <div class="pitfall-why">原因：== 比较的是引用地址，不是内容。</div>
  </div>
</div>
```

踩坑区规则：
- 每个坑必须有 ❌ 错误写法 + ✅ 正确写法 + 原因解释
- 只放真正高频的坑，不要凑数
- 一个模块 1~5 个坑点为宜

### 代码示例格式

所有代码块统一用以下格式，**必须包含复制按钮**：

```html
<pre>
<button class="copy-btn" onclick="copyCode(this)">复制</button>
<span class="lang-tag">Java</span>
<span class="kw">public class</span> <span class="type">MyClass</span> {
    <span class="cmt">// 注释</span>
    <span class="kw">private</span> <span class="type">String</span> name;
}
</pre>
```

```html

</div>
```

## 多文件处理

如果用户一次上传多个文件：
1. 分别读取每个文件
2. 按文件名/内容主题识别模块归属
3. 合并到同一个 HTML 手册中
4. 每个文件对应一个或多个导航模块

## 快捷键

生成的手册必须支持以下键盘快捷键（JS 实现）：

| 快捷键 | 功能 |
|--------|------|
| `Ctrl+K` / `Cmd+K` | 聚焦搜索框 |
| `Escape` | 关闭搜索 / 关闭移动端菜单 |
| `T` | 切换暗亮主题（仅在输入框未聚焦时） |

## 质量标准

生成 / 更新的手册必须满足以下标准：

- [ ] 所有 `{{变量}}` 已替换为实际内容
- [ ] 导航链接与 section id 一一对应
- [ ] 侧边目录条目与 section 一一对应
- [ ] 学习路线图与模块顺序一致
- [ ] 每个代码块都有语言标签和复制按钮
- [ ] 每张卡片都有难度标签和来源标注
- [ ] 有前置依赖的卡片已标注 `.prereq`
- [ ] 易错点已整理到踩坑区（.pitfall）
- [ ] 对比内容用表格呈现，流程内容用步骤图
- [ ] HTML 为单文件，不依赖外部资源（无 CDN 引用）
- [ ] 暗亮主题切换正常工作
- [ ] 搜索功能可搜索所有正文内容（Ctrl+K 可聚焦）
- [ ] 快捷键正常工作（Ctrl+K / Esc / T）
- [ ] section 可折叠/展开
- [ ] 变更日志存在且最新记录在最上面
- [ ] 打印时导航栏/侧边栏隐藏、背景色去除
- [ ] 移动端响应式布局正常
- [ ] 页脚包含生成日期
- [ ] 内容忠实于原始材料，不编造
- [ ] （迭代模式）已有内容完整保留，无丢失

## 注意事项

- 不要编造内容，只整理用户提供的材料
- 保留原始代码的完整性，不要删减代码
- 如果原文有图片引用，用 ASCII 图或文字描述替代
- 生成的 HTML 必须是单文件，不依赖外部资源
- 中文内容优先，标题和说明用中文
- 导航栏模块数量控制在 3~12 个
- 如果内容过多，做合并而非拆分更多模块
- **迭代时：宁可多检查一遍，也不要丢失已有内容**
- **每张卡片必须标注难度和来源**
- **踩坑区只放真正高频的坑，不要凑数**
- **变更日志只增不删，记录手册的完整演进历史**
